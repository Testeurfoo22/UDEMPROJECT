2) Discussion

Par rapport au devoir 2, la conception a été grandement améliorée surtout au niveau de dépendre d'interfaces et de classes abstraites au lieu de classes directement (respecte le DIP). C'était déjà un départ dans le devoir 2, mais c'était fait de manière non idéale ne respectant pas nécessairement comment une classe abstraite fonctionne et un problème majeur était comment les instances étaient créées, car ça nécessitait plusieurs informations pouvant dépendre de différentes classes. Avec le patron de fabrique, ce problème a pu être résolu. Parlant du patron de fabrique, une faiblesse de ce patron est que c'est difficile d'ajouter un nouveau type d'élément après que le patron ait été fait. C'est pour cette raison que nous avons ajouté les véhicules comme classe, dans l'éventualité 100% théorique que traiter cela serait bon pour les besoins de notre application. Ainsi, cela fait aussi que notre application traite une possibilité de plus par rapport au devoir 2.

Lors du devoir 3, nous avons aussi plus clairement défini comment l'application était accédée par les utilisateurs, que ce soit les clients ou les administrateurs. Le patron de l'observateur a apporté cette réflexion, car il fallait se demander comment les données sont accédées selon les types d'utilisateur ainsi que comment elles sont sauvegardées. Ce n'est pas le genre de questions qu'on s'était posées lors du devoir 2, par exemple.

3) Java

Pour notre logiciel .java, on a du utiliser les 6 patrons suivants, fabrique, singleton, etat, commande, observer, visiteur. Pour la fabrique et singleton, on a 3 duos de fichier, abstract et factory, pour les localisations, les companies et les vols, puisque ces derniers ont les memes variables. pour ce faire, on a ajouté certaintes variables pour les croisieres et train ou les companies de ces derniers, avec l'ajout de code iata pour une meilleure visibilite des id. dans ces fichiers, on a une instance, representant toutes les données d'un type, du plus, on a les getters setters utiles pour la manipulation, avec aussi les fonctions create/update, delete n'est pas present car en java,car normalement ne plus avoir un acces a une adresse la supprime (pas de malloc donc on supprime directement de notre liste). Nos sieges sont creer a partir d'une section (enum qui varie en fonction du type de voyage), d'une taille, et d'un etat, qui permet de savoir si ce dernier est reserve/paye. Les commands/observer/visiteur permettent de gerer les demande d'acces aux commandes de recherche/ajout/modification/suppression d'objects, avec la differentiation entre client et admin. Notre Interface, fonctionne comme suit, la page systeme est la premiere page qui est crée par le programme, on peut voir ceci comme un serveur, d'ici on peut creer nos instances admins/clients, chaque page represente une personne. Etant donne que les observer sont lié par nos instances, meme si ces derniers sont uniques et propre a chacun, Si une modification est effectué par un admin (ajout/update/delete) ou par un client (reserveation/inverse), tout le mode pourra voir la modification
